【AITeach 答辩关键追问 QA】

Q1：你们这个项目到底是做什么的？
A：我们做的是一个面向电机学课程学习场景的智能教学辅助平台。它不是通用聊天机器人，而是围绕电机学学习中的几个核心痛点来设计的，包括题目不会做、知识点难理解、错题难整理、练习后难复盘、仿真门槛高。系统把 AI 解题、课程问答、错题本、AI 错题教练和仿真实验整合到一个 Web 平台里，目标是形成“解题—纠错—分析—练习—仿真理解”的完整学习闭环。

Q2：你们的项目创新点是什么？和普通 AI 问答网页有什么区别？
A：我们的创新点不在于“接了一个大模型”，而在于把大模型放进了电机学这个垂直课程场景里。普通 AI 网页更多是一次性问答，我们这个系统更强调持续学习过程。第一，它是课程定向的，知识和解题风格都围绕电机学。第二，它不是只回答一道题，而是把错题沉淀、薄弱点分析、举一反三练习和仿真实验串起来。第三，它从最初原型继续演进成了 Web 化产品形态，具备流式回答、公式渲染、错题管理和可分发启动的能力，所以更接近真实教学工具，而不是简单 demo。

Q3：你们为什么选电机学这个方向？
A：因为电机学是典型的高门槛课程，特点是公式多、概念抽象、题目综合性强，而且学生往往不仅是“不会做题”，还会在错题分析、知识迁移和实验理解上卡住。我们选这个方向，是希望把 AI 的能力用于一个真实存在且比较难学的专业课程场景，这样项目的应用价值更明确，也更能体现垂直智能教学的意义。

Q4：这个系统的总体架构是怎样的？
A：总体上是前后端分层架构。前端是原生 Web 页面，负责交互、消息展示、公式渲染和页面切换。后端是 FastAPI，负责所有业务接口和系统编排。后端下面再分成几层：一层是 services，负责 AI 解题、问答、错题分析、仿真助教等业务逻辑；一层是 storage，负责错题本 SQLite 数据读写；一层是 simulation，负责电机学仿真计算和 MATLAB 对接；还有 schemas 负责数据结构约定。这样拆分的好处是界面、业务、存储、仿真彼此职责清楚，后续扩展和维护都更方便。

Q5：AI 解题这一块是怎么工作的？
A：AI 解题分文字题和图片题两种入口，但最后都会进入统一的解题链路。用户在前端输入题目或上传图片后，前端把题目、上下文和附件发给后端。如果是图片题，后端会先做 OCR 或视觉模型识别，提取题干。之后系统会结合电机学知识库做召回，把相关知识点和题目一起组织进 Prompt，再交给大模型进行推理和作答。最后后端把结果流式返回前端，前端边接收边展示，并支持公式渲染。这样做的目的，是尽量减少模型在专业场景里的胡乱回答，提高电机学题目的准确率和规范性。

Q6：你们为什么不直接做一个普通聊天框，而要做流式回答和公式渲染？
A：因为电机学题目天然包含公式、推导和分步骤分析。如果只是一次性返回纯文本，体验会比较差，也不利于教学。流式回答能让用户看到模型逐步输出，交互感更接近真实对话；公式渲染则能保证数学表达是可读的，而不是一堆原始符号。我们在前端还专门处理了流式场景下公式分隔符不完整的问题，避免公式在输出过程中闪烁或乱码。这些设计不是为了炫技，而是为了让专业学习场景更自然。

Q7：错题本和 AI 教练是怎么形成闭环的？
A：当用户在 AI 解题页面遇到做错的题，可以把题目加入错题本，并补充自己的错误原因。错题本会长期保存题干、参考答案和错因。AI 教练会基于这些错题记录去分析用户的薄弱点，比如某一类知识点掌握不稳、某一类错误反复出现，然后给出学情分析和学习建议。在此基础上，它还能自动生成一题针对性的练习题。用户查看解析后，还可以选择把这道练习题加入错题本，继续沉淀。这样系统不是只回答问题，而是帮助学生复盘、总结和再次练习。

Q8：为什么 AI 教练出的练习题不是自动加入错题本，而是用户选择加入？
A：这是我们有意识做的产品设计。真实做错的题，加入错题本是很自然的；但 AI 教练生成的练习题，不一定每道都应该直接记成错题，所以我们保留了用户选择权。用户可以先看解析，再决定是否把这道题加入错题本。这种设计一方面避免无效记录过多，另一方面也更符合学习过程中的自主复盘逻辑。

Q9：仿真模块是怎么设计的？为什么要做仿真？
A：我们认为电机学不仅是会做题，还要理解电机运行规律和参数变化对结果的影响，所以加入了仿真实验模块。现在项目围绕四类典型对象组织实验，分别是直流电机、变压器、异步电机和同步电机。用户可以在页面里选择实验，设置参数，运行后得到关键指标、数据序列、结果解释和图形化展示。仿真的作用不是替代 MATLAB 专业软件，而是降低理解门槛，让学生在学习过程中更容易把题目、公式和物理过程联系起来。

Q10：你们为什么还接 MATLAB？Python 不是也能算吗？
A：这是一个很关键的问题。确实，简单数值计算 Python 也能完成，所以 MATLAB 的意义不在于“重复算一遍”，而在于更接近工程仿真场景。我们目前采用的是分层策略：基础计算可以由 Python 完成，保证系统轻量、稳定、易分发；而 MATLAB 是可选增强链路，用于更高保真或更工程化的仿真能力。特别是仿真结果不只是数值，也可以输出图像或更完整的过程结果，这样更适合展示和教学。也就是说，我们不是为了“用 MATLAB 而用 MATLAB”，而是把它放在真正有价值的位置上。

Q11：如果别人电脑没有 MATLAB，项目还能不能用？
A：能用。我们一开始就考虑了这个问题，所以 MATLAB 不是必需依赖，而是可选能力。系统默认可以在没有 MATLAB 的情况下运行基础仿真，保证普通同学也能直接使用。只有在本机已经安装 MATLAB，并且希望使用更高保真链路时，才需要额外配置 MATLAB CLI 路径。这样做的好处是，既保留了“真 MATLAB”能力，又不会把整个产品绑死在 MATLAB 环境上。

Q12：你们为什么没有直接做公网部署，而是做成本地 Web 应用？
A：因为我们当前的项目目标是课程结题和产品原型验证，不是立即做商业化运营。公网部署会牵涉服务器、域名、HTTPS、账号管理、并发、安全、模型费用等很多额外问题，这些投入对当前阶段的收益不高。我们更合理的策略是，先把本地 Web 原型做到完整、稳定、可分发、可演示，让老师和用户能清楚看到产品价值。等后续需要真正扩展时，再把它迁移到服务器上，这个架构本身是支持继续上线演进的。

Q13：你们怎么保证 AI 解题的准确率？
A：准确率是这个项目最核心的问题，所以我们没有把它当成单纯的通用大模型问答来做。我们做了三层工作。第一层是课程场景约束，把解题风格固定在电机学专业语境内。第二层是知识增强，也就是在解题前尽量提供相关知识内容，降低模型胡说八道的概率。第三层是针对电机学任务做模型优化和微调，提升专业题目的稳定性和正确率。我们的目标不是让模型“像人闲聊一样自然”，而是让它“像一个会解电机学题目的助教一样可靠”。

Q14：你们做这个项目最大的技术难点是什么？
A：我认为有三个主要难点。第一个是专业准确率，因为电机学题目对概念和公式要求很高，大模型稍不注意就会漏条件或者乱推导。第二个是前端交互体验，尤其是流式回答和公式渲染，专业内容一旦展示不好，用户体验会明显下降。第三个是系统整合，因为我们不是一个单点功能，而是要把 AI 解题、错题本、AI 教练、仿真和 Web 页面串成完整系统。也正因为这些难点都存在，这个项目更像一个完整产品原型，而不是简单拼接。

Q15：你们为什么从 Streamlit 迁移到真正的 Web 架构？
A：因为 Streamlit 更适合快速验证原型，但不太适合我们后期这种强调页面结构、交互细节、流式聊天、错题闭环和可分发形态的产品目标。前期用 Streamlit 是为了快，后期迁移到 FastAPI 加原生前端，是为了把系统做得更接近真实 Web 产品。这个迁移本身也体现了项目从“功能验证”走向“工程化原型”的过程。

Q16：你们项目现在大概是什么规模？
A：它已经不是小 demo 了。按我们目前统计，排除虚拟环境和缓存后，项目大约是四十多个代码和文档文件，总行数在九千行左右，属于本科生单人主导开发里比较扎实的中等规模项目。更重要的是，它不是靠单个大文件堆出来的，而是已经有前后端分层、业务模块划分、存储层和仿真层的结构。

Q17：如果继续做下去，你们下一步最想完善什么？
A：如果后续继续推进，我最想完善三件事。第一是进一步提高专业题目的准确率和稳定性，让 AI 解题更可靠。第二是把仿真结果进一步图表化和过程化，让学生看到更完整的动态曲线和实验现象。第三是继续向正式 Web 产品演进，比如更细的模块拆分、更完善的用户系统，以及真正的服务器部署能力。也就是说，现在这个项目已经完成了高质量原型，后续重点会是从“可用”走向“更成熟”。

Q18：如果老师怀疑这是不是你们自己做的，怎么回答？
A：最稳的回答不是空口说“是我们做的”，而是直接讲清架构和决策。可以这样答：我们项目前端是原生 Web，后端是 FastAPI，AI 解题、错题分析和仿真分别走不同服务链路，错题数据用 SQLite 存储，仿真部分保留了 Python 和 MATLAB 的双路径策略。前期我们用 Streamlit 快速验证功能，后期迁移到 Web 形态，是因为需要更好的交互和产品化展示。只要能把这些设计原因讲清楚，老师一般就能判断出你们对项目是真正理解的。

【答辩时最建议背熟的 3 句总述】
1. 我们不是做一个通用 AI 聊天工具，而是做一个面向电机学课程的智能教学辅助平台。
2. 这个系统最核心的价值，是把解题、错题、分析、练习和仿真串成了一个完整学习闭环。
3. 项目已经从原型验证发展为具备 Web 化交互、稳定运行和可分发能力的产品原型。

【最后建议】
答辩时不要一上来就讲模型名和技术名词，先讲“痛点、闭环、价值”，再讲架构和实现。老师通常先判断“这个项目有没有意义”，然后才会判断“你们做得深不深”。
